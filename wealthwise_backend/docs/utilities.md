# Utility Endpoints Documentation

## Overview

Additional utility endpoints for health checks, data seeding, and system operations. These endpoints provide supporting functionality for the main API.

## Endpoints

### 1. Health Check

**Endpoint:** `GET /api/health/`

**Permission:** Public (AllowAny)

**Description:** Check the health status of the API and its dependencies (database connection).

**Response (200 OK):**
```json
{
    "status": "healthy",
    "timestamp": "2024-01-21T14:30:00.123456Z",
    "database": "healthy",
    "version": "1.0.0",
    "services": {
        "api": "up",
        "database": "healthy"
    }
}
```

**Response (503 Service Unavailable) - Database issue:**
```json
{
    "status": "degraded",
    "timestamp": "2024-01-21T14:30:00.123456Z",
    "database": "unhealthy: connection refused",
    "version": "1.0.0",
    "services": {
        "api": "up",
        "database": "unhealthy: connection refused"
    }
}
```

### 2. Seed Historical Data

**Endpoint:** `POST /api/seed-data/`

**Permission:** IsAuthenticated

**Description:** Generate realistic historical financial data for testing and demo purposes. Creates accounts, transactions, budgets, goals, alerts, and settings with data spanning multiple years.

**Request Body:**
```json
{
    "years": 5
}
```

**Parameters:**
| Name | Type | Required | Default | Description |
|------|------|----------|---------|-------------|
| years | integer | No | 5 | Number of years of historical data to generate |

**Response (200 OK):**
```json
{
    "status": "success",
    "message": "Successfully seeded 5 years of historical data",
    "data": {
        "accounts_created": 6,
        "budget_categories_created": 8,
        "goals_created": 6,
        "transactions_created": 2450,
        "expenses_created": 735,
        "alerts_created": 5,
        "period_covered": "2019-01-21 to 2024-01-21"
    }
}
```

**Warning:** This operation will:
- Delete all existing data for the current user
- Create new realistic financial data
- Generate 5 years of transaction history with seasonal patterns

## Example Usage

### Check API Health
```bash
curl -X GET http://localhost:8000/api/health/
```

### Seed 3 Years of Data
```bash
curl -X POST http://localhost:8000/api/seed-data/ \
  -H "Authorization: Bearer <access_token>" \
  -H "Content-Type: application/json" \
  -d '{
    "years": 3
  }'
```

### Seed Default (5 Years)
```bash
curl -X POST http://localhost:8000/api/seed-data/ \
  -H "Authorization: Bearer <access_token>" \
  -H "Content-Type: application/json"
```

## Use Cases

### Health Check
- **Monitoring:** Use for uptime monitoring and alerting
- **Load Balancers:** Health check endpoint for load balancer configuration
- **CI/CD:** Verify deployment success in pipelines
- **Frontend:** Check API availability before making requests

### Seed Data
- **Demo Accounts:** Create realistic demo data for presentations
- **Testing:** Generate test data for development and QA
- **Onboarding:** Help new users understand the app with sample data
- **Performance Testing:** Generate large datasets for load testing

---

## Rate Limiting

The seed endpoint is resource-intensive. Consider implementing:
- Rate limiting (e.g., max 1 request per hour per user)
- Admin-only access in production
- Async processing for large datasets

---

## Security Considerations

1. **Health Check:** Safe to expose publicly - no sensitive data
2. **Seed Data:** 
   - Deletes user data - requires confirmation in production
   - Should be restricted to development/testing environments
   - Consider requiring admin privileges

## Integration Examples

### Docker Health Check
```dockerfile
HEALTHCHECK --interval=30s --timeout=10s --start-period=5s --retries=3 \
  CMD curl -f http://localhost:8000/api/health/ || exit 1
```

### Kubernetes Probe
```yaml
livenessProbe:
  httpGet:
    path: /api/health/
    port: 8000
  initialDelaySeconds: 30
  periodSeconds: 10
```
